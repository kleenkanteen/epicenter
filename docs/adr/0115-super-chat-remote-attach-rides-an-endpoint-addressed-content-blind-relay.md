# 0115. Super Chat remote attach rides an endpoint-addressed, content-blind relay

- **Status:** Accepted (wave 1 implements plaintext loopback forwarding; Cloud attach seals by default and does not ship before the sealing wave)
- **Date:** 2026-07-07
- **Relates:** [ADR-0080](0080-the-super-app-is-a-desktop-host-cross-device-is-remote-access-to-the-session-not-a-per-app-capability-plane.md) (resolves its open product question: Epicenter operates this relay for turnkey mobile remote), [ADR-0079](0079-cross-device-is-two-planes-epicenter-syncs-the-crdt-the-box-is-reached-directly.md) (this is the "future hosted remote" its 2026-07-02 update names, and it is not the deleted capability plane), [ADR-0086](0086-no-live-consumer-for-network-reachable-capability-reach-opensidian-is-superseded-not-migrated.md) (network-reachable capability reach stays refused; a session channel is not a capability route), [ADR-0073](0073-tools-speak-mcp-natively-epicenter-owns-only-the-transport-mcp-lacks.md) (the relay-floor organs stay deleted; this does not resurrect them), [ADR-0113](0113-super-chat-session-commands-are-host-owned-transports-only-frame-them.md) (Super Chat session commands are host-owned; the relay only forwards their sealed bytes), [ADR-0004](0004-trust-the-relay-reject-zero-knowledge.md) and [trust-model](../trust-model.md) (the sync anchor holds plaintext; this relay forwards sealed frames it cannot read), [ADR-0068](0068-privacy-is-a-deployment-not-a-product-feature.md) (privacy is the deployment: self-host makes the operator the user), [ADR-0075](0075-self-host-is-a-single-partition-instance-behind-one-operator-supplied-bearer.md) (a self-hosted instance is reached by URL and token)

## Context

Super Chat is a desktop host that owns local tools and local source data; a phone or a second browser is a thin client that attaches to the one running session (ADR-0080, ADR-0113). The product pull is that a signed-in phone reaches that session with no URL, no tailnet, and no manual setup. The desktop is NAT'd, so reaching it needs a publicly addressed rendezvous both sides dial out to (ADR-0079), and the rendezvous the phone already trusts is Epicenter itself. Epicenter once shipped a per-user relay that routed named app routes across devices (the relay floor, ADR-0073), and it was deleted because no consumer needed network-reachable per-app capability reach (ADR-0086, PR #2277). The open question is the abstraction level of a hosted remote that does have a live consumer: Super Chat remote attach.

## Decision

Remote Super Chat attach rides an **AttachRelay**: Epicenter forwards sealed bytes between two authenticated endpoints of one principal, a signed-in client endpoint and a desktop Super Chat host endpoint.

1. **The relay is endpoint-addressed, never route-addressed.** It routes by `principalId`, `hostId`, `deviceId`, and `attachId`. The routing unit is a pair of endpoints under one principal, not a named route. There is no route name, no route registry, no `exposedRoutes` presence, no client auto-mount, no MCP carried as a platform surface over the relay, no CLI `tools` or `call` surface, and no per-app endpoint directory. Those are the relay-floor organs deleted in PR #2277, and endpoint addressing is what keeps them from having anywhere to live: with no route namespace, there is nothing for a capability registry to populate. The AttachRelay is not the relay floor.

2. **The relay holds no keys and reads no frames.** It forwards opaque sealed bytes. Super Chat owns the session protocol (ADR-0113): the client commands and host snapshots are sealed by the two endpoints and are never parsed by the relay. What the relay may know is the envelope only: `principalId`, `hostId`, `deviceId`, `attachId`, and timing and byte counts. What it is forbidden to know is everything inside the seal: prompt text, streaming chunks, tool results, approval answers, source excerpts, MCP methods, and the Super Chat command types themselves.

3. **Pairing, device grants, and key agreement belong to the account and device layer, not the relay and not Super Chat.** A device pairs once (a QR scan or an account-mediated challenge), the desktop approves it and mints a revocable per-device grant, and the two endpoints agree on a session key the relay never sees. This layer also owns the host directory that lets a signed-in phone discover that a Mac has a live Super Chat host, carrying only `hostId`, a label, and liveness status, never a list of what the host exposes. With one consumer, an attachable host is a Super Chat host by definition, so no capability field is needed; adding one would be `exposedRoutes` in a closed-enum costume, and it is refused until a second consumer earns it.

4. **The product surface is `attachToSuperChatHost(hostId)`, and it is the only consumer.** The byte-forwarding primitive is generic only in the narrow sense of clause 1, that it forwards sealed bytes between two endpoints. It is not exported from `packages/workspace` as a framework verb, and it is used only by `packages/server` and `apps/super-chat`. A second consumer requires a new ADR naming a real, live consumer, the same bar ADR-0086 set.

5. **Confidentiality follows the deployment, and Cloud always seals.** On Cloud, attach frames are always sealed, so the relay forwards ciphertext it cannot read. Sealing uses key agreement authenticated to the device grant, so the Cloud relay cannot man-in-the-middle the channel and read the frames. This keeps the whole live session, including the tool-result stream, off the operator plane, the line the seam-2 rule and ADR-0080 already draw. Sealing is not gated on classifying a frame as sensitive: there is no sensitivity classifier, because Cloud seals unconditionally. On self-host the operator is the user (ADR-0068, ADR-0075), so plaintext is an explicit, fail-closed opt-out of the sealing default, and only a self-host operator can take it: topology, not client crypto, is the privacy answer (ADR-0004). The relay code is identical in both; sealing is a layer above it, on by default.

6. **Transport posture.** The AttachRelay is a WebSocket channel on the per-user rendezvous each device already dials out to, because a Durable Object is a WebSocket router and the cloud cannot be a UDP peer (the reason ADR-0073 deleted iroh). Tailscale stays a developer proof for reaching a loopback host, never the product transport. An iroh direct path stays a deferred optimization behind a measured trigger, never the product's spine.

## Consequences

- One sentence explains it: Epicenter forwards sealed bytes between two of your own signed-in devices, addressed by device, never by route.
- "Phone just works after Epicenter sign-in" holds on Cloud and self-host identically, because the rendezvous is the same Epicenter instance the phone already signs into (a URL and a token on self-host, ADR-0075).
- The relay-floor organs stay deleted: the route table, the MCP gateway catalog, `exposedRoutes` presence, the channel router's route addressing, and the CLI `tools`, `call`, and `--relay-expose` verbs. PR #2277's presence-schema guard that rejects route-shaped and action-shaped fields stays in force, and the AttachRelay directory must pass it.
- The single per-launch shared Super Chat token is replaced by per-device grants. ADR-0113 is untouched: the transport still only frames the host-owned commands, and now the transport is the AttachRelay.
- Sealing is a hard prerequisite for Cloud attach: Cloud attach does not ship before the authenticated sealing layer lands, because until it does the content-blind claim is false (an unauthenticated key exchange over the relay could be man-in-the-middled). Self-host attach can ship first on plaintext, since the operator is the user. There is no plaintext-Cloud-beta option.
- The anti-MITM secret is a distinct pairing secret, not the device grant. "Key agreement authenticated to the device grant" (clause 5) is refined by the sealing proof: the device grant authenticates the socket *to the relay* (the client presents it as `bearer.<grant>` and the mount hashes and compares it, so the relay sees the raw grant transiently on every connect), and a secret the relay sees cannot defeat a malicious relay. So the exchange is authenticated by a *pairing pre-shared key* the relay never sees, an out-of-band artifact of the same pairing (the QR/paste of clause 3). Two questions, two secrets: the grant answers "may this socket connect," the PSK answers "is my peer the device I paired with." Deriving both from one pairing secret (so the relay-scoped grant and the peer-scoped PSK share a root) is an account-layer refinement, not required for content-blindness.
- The sealing mechanism is proven and lives in the Super Chat adapters, above an unchanged relay: fresh ephemeral ECDH (P-256) per session for forward secrecy, PSK-keyed HMAC key-confirmation over the transcript to reject a substituted key, and per-direction XChaCha20-Poly1305 with counter nonces. The relay coordinator (`packages/server/src/attach-relay/core.ts`) forwards the sealed frames as opaque `payload` bytes and never imports the sealing module, so it stays byte-, key-, and frame-blind. Plaintext remains the self-host opt-out (sealing absent on the adapter).
- This forecloses a second product consumer without an ADR, a route-addressed capability plane, and any reading of Cloud session frames once they are sealed.

## Considered alternatives

- **A. A Super Chat-only relay that routes Super Chat session frames.** Rejected as incoherent with content-blindness: a relay that cannot read the frames cannot be Super Chat aware, so "Super Chat only" describes the product surface, not the mechanism. The mechanism is endpoint-addressed byte forwarding, and the narrowness lives in the product surface (clause 4), which is where this decision puts it.
- **B. A generic device message relay whose product API routes arbitrary sealed frames between signed-in devices, with Super Chat as one consumer.** Refused. A general routing product surface regrows a route namespace, presence-advertised routes, and per-app endpoints, which is the relay floor ADR-0086 retired. Genericity is confined to the byte-forwarding mechanism and is never exposed as a routing product.
- **A plaintext hosted broker for Cloud.** Rejected: the session carries tool results (iMessage, mail, books, files), so a content-readable broker re-leaks exactly the data the seam-2 rule keeps off the operator plane (ADR-0080). Plaintext is admissible only where topology makes the operator the user (self-host), as a fail-closed opt-out.
- **Iroh as the product transport.** Deferred: it needs a mobile SDK and Epicenter-operated production relays, its browser story is relay-only, and its direct path relays anyway for a cellular phone reaching a home NAT. It returns only behind a measured trigger.
- **Tailscale as the product transport.** Rejected: bring-your-own-overlay is a developer proof, and a browser off the tailnet needs Funnel (beta, throttled, admin-gated). It proves the shape; it is not the product.
