# 0100. AI credits are product-priced units, and the charge shape follows whether cost is known before the call

- **Status:** Accepted
- **Date:** 2026-07-02
- **Relates:** [ADR-0054](0054-an-inference-backend-is-the-metered-gateway-or-a-custom-server.md) (the metered hosted gateway these credits price; house-key-only, no BYOK bypass), [ADR-0053](0053-the-epicenter-bearer-is-an-audience-scoped-credential.md) (the audience-scoped bearer that authorizes metering)

## Context

The hosted gateway meters every AI call ([ADR-0054](0054-an-inference-backend-is-the-metered-gateway-or-a-custom-server.md)), but "meter" was never pinned to a unit. Two failure modes bracket the design. Request-unit billing (one credit per call regardless of work, the model GitHub Copilot is retreating from) prices a two-second clip the same as a thirty-minute recording. Raw provider pass-through (bill the user Autumn's cents-per-token) makes spend unpredictable and leaks provider pricing into the product. Between them sits the real constraint: some AI costs are known before the call (a chat model has a fixed per-call credit price) and some are not (a transcription's cost is its audio duration, known only after the provider returns). A billing shape that ignores that split either over-charges cheap calls or cannot fail closed on expensive ones.

## Decision

An Epicenter AI credit is a predictable product unit, not a request count and not a provider cost. All paid AI usage shares **one billing lifecycle against one credit wallet**: authorize before the provider runs, run the provider, then settle successful usage. Chat and STT are the same policy at two different cost-knowledge states; what differs is only the *authorize* step, and only because one call's cost is known before the provider runs and the other's is not. The durable direction is a shared lifecycle with distinct authorization, not identical mechanics.

- **Authorize (chat): reserve the exact amount.** Cost is known before the call, so the model's catalog credit price is taken as an Autumn lock before the provider call. The gate is exact: a wallet that cannot cover this model's price fails closed before any provider spend.
- **Authorize (STT): require a usable wallet.** Cost is not reliably known before the call (a transcription's cost is its audio duration, returned only after the provider answers), so a cheap check refuses an empty wallet (or a provider outage that hides the balance) but cannot prove "enough for this call." Transcription is priced per transcribed minute (rounded up, floor of one credit).
- **Settle (both): charge the wallet on success; authorization fails closed, settlement never fails the response.** If authorization cannot be verified before spend, the call is rejected (`failOpen: false`). On success, chat confirms its lock and STT tracks the real per-minute charge from the duration the gateway returns, both off the after-response queue. Once the provider has answered the user, settling is best-effort telemetry: its failure is swallowed, never surfaced as an error to a browser that already holds its result. (Chat also releases its lock on a pre-stream failure; the lock's TTL auto-releases if the worker dies before settling.)

The lifecycle is shared; the reservation shape is not. STT is deliberately not a fixed per-call reservation, and forcing it into chat's reserve-exact lock is rejected below.

## Consequences

- **One lifecycle, two authorization shapes, not one symmetric mechanic.** Chat and STT share the authorize-run-settle lifecycle and one credit wallet, but authorize through different code paths (`reserveAiChat` with a lock versus `checkAiCredits` plus `trackAiTranscription`) because their cost-knowledge differs. A later reader should unify the *lifecycle* language, never collapse the two authorization shapes into a single lock.
- **STT's pre-gate is structurally weaker than chat's, and that is inherent.** Because duration is unknown up front, the STT gate can only prove "some credits exist," not "enough for this call," so a successful call can settle a charge larger than the balance and tip the wallet negative. The overspend is bounded by two things, not one: the largest single call (one long recording can over-tip a near-empty wallet on its own, with no concurrency at all) and the number of in-flight calls that each pass the gate before any usage posts. The named future remedy, taken only if abuse or material overspend appears, is an **input duration/size ceiling before transcription**, not an STT reservation lock. A byte-size ceiling is the cheap first guard (file size is known before the call), but it is an imperfect proxy: a small compressed file can still be long, so the honest control is a duration ceiling, which may require probing container metadata or a decode preflight. A reservation lock is explicitly not the default remedy: it would force the duration preflight no comparable provider performs and would break the shared-lifecycle-with-distinct-authorization shape this ADR settles on.
- **The check path depends on Autumn parsing balance responses that legitimately contain nulls.** A metered feature that only feeds a credit system (`ai_usage` into `ai_credits`) reports `ai_usage: null` in `check` responses. Both pre-provider gates read those balance maps: chat's `reserveAiCreditsWithLock` and STT's `checkAiCredits` each call `autumn.check`. `autumn-js` must therefore be at a version whose response schema treats those map values as nullable (1.2.33 or later; the repo pins `^1.2.34`); below that, a `check` that returns a null `ai_usage` balance throws a client-side `ResponseValidationError`. This pin is load-bearing for the check path and must not be reverted. The STT post-success `track` runs `async: true` (see below) and returns no balance body, so it never exercises this surface, but that does not remove the need for the pin.
- **Post-success STT metering is an async usage event, not a synchronous `track` (shipped).** Settling is best-effort and tolerant of eventual consistency, so the STT charge is enqueued with `async: true` (`service.ts`, `trackAiTranscription`): Autumn records the event and answers 202 with no `balances` body, so the post-success path has no response-validation surface at all. This is a metering-path semantics choice, not a bug fix: the nullable schema retained for the check path above is what actually prevents the `ResponseValidationError`. Chat's lock finalize is unaffected and stays synchronous.
- **Provider pricing stays out of the product.** Users see credits, never tokens or cents. The credit-to-provider-cost margin is a catalog concern (`ai-model-pricing.ts`), invisible at the billing boundary.

## Considered alternatives

- **Fixed per-call STT reservation** (one credit reserved and confirmed per transcription, mirroring chat's lock). Rejected: it makes "fail closed before spend" clean but prices a thirty-minute recording identically to a two-second clip, which is the request-unit problem this ADR exists to avoid. Accurate product units beat a tidy symmetric gate.
- **Raw provider pass-through** (bill Autumn's per-token or per-minute provider cost directly). Rejected: unpredictable user-facing spend, and provider pricing leaking into the product; credits exist precisely to decouple the two.
- **One unified metered call *shape* for chat and STT** (the same reservation mechanic for both, not just the same lifecycle). Rejected: chat cost is known pre-call and STT cost is not, so a single mechanic must either over-charge cheap calls or drop the pre-spend gate. The shared lifecycle (authorize, run, settle) is accepted; a shared *authorization mechanic* is the part that is rejected.
- **Strict prepaid with an STT duration reservation** (probe or worst-case reserve so the wallet can never go negative). Rejected: no comparable provider (OpenAI, OpenRouter, GitHub Copilot, Metronome, OpenCode) reserves exact cost for a variable-cost call; all gate on a usable balance and meter after, accepting a bounded overspend. The one comparator that reserves before the call (T3 Chat) does so for an *estimable chat* cost, which is exactly Epicenter's chat case, not its STT case. Duration preflight buys a marginally tighter gate at the cost of latency and blocked legitimate long recordings; the size/duration ceiling above is the cheaper, honest bound.
