# 0060. An inference connection is a base URL and an optional bearer key; everything non-static is an injected transport

- **Status:** Accepted
- **Date:** 2026-06-24
- **Amends:** [ADR-0059](0059-an-inference-connection-is-a-capability-orthogonal-device-endpoint.md) (collapses the hosted/custom `kind` discriminant into one flat connection; specifies the model-id tie-break and that capability metadata is read-side decoration)
- **Relates:** [ADR-0053](0053-the-epicenter-bearer-is-an-audience-scoped-credential.md) (the audience-scoped bearer that makes the injected hosted transport safe), [ADR-0054](0054-an-inference-backend-is-the-metered-gateway-or-a-custom-server.md) (the metered house-key gateway hosted points at), [ADR-0050](0050-the-inference-contract-is-openai-compatible.md) (the OpenAI-compatible wire every connection speaks), [ADR-0022](0022-rust-owns-the-models-folder-the-webview-owns-the-catalog.md) (the downloaded-binary engine kind that is *not* a connection)

## Context

[ADR-0059](0059-an-inference-connection-is-a-capability-orthogonal-device-endpoint.md) modeled an inference connection as `{ kind: 'hosted' } | { kind: 'custom'; preset?; baseUrl; apiKey? }`. Two seams show the `kind` discriminant is not the real axis:

1. Client-side, hosted and custom run the identical operation: POST an OpenAI-compatible body to a base URL with some credential, stream the reply back. Hosted already flows through the registry as an *injected* `{ fetch, baseURL }` (the per-app `auth.fetch` plus the gateway URL); `{ kind: 'hosted' }` is only a marker that says "return that injected transport." The genuine differences (metering, the house key, the curated catalog) are server-side.
2. The "uniform Bearer" assumption is correct for every shipped preset and, a code-grounded review confirmed, for every OpenAI-compatible chat provider in practice, including the OpenAI-compatibility endpoints that Anthropic, Gemini, and Bedrock-with-an-API-key now ship: all authenticate with `Authorization: Bearer`. The review also surveyed how mature tools (opencode, Continue, Zed, the Vercel AI SDK `openai-compatible` provider, LiteLLM, `llm`) model this primitive: every one collapses auth into "an optional key sent as a Bearer header, plus a free-form headers bag for the rare rest," and not one uses an auth-strategy discriminant.

A review also confirmed a billing hazard: the hosted catalog sells real upstream ids (`gpt-5.5`, `packages/constants/src/ai-providers.ts`), so a user's own OpenAI key collides with hosted on the same id.

## Decision

An inference connection is one device-local primitive: a base URL and an optional bearer key. There is no `kind` discriminant, no auth-strategy union, and no headers bag.

```ts
type Connection = { baseUrl: string; apiKey?: string };
resolveConnection(connection: Connection): { fetch, baseURL } // one branchless rule, no hosted argument
```

- **`apiKey` is sugar for `Authorization: Bearer`.** `resolveConnection` attaches the key as a Bearer when present; a keyless server gets a bare fetch. A provider difference is data (a base URL), never a code branch.
- **Hosted is not a member of this type; it is the registry's injected transport.** The device holds a set of `Connection`s (zero or more) plus one injected hosted `ResolvedConnection` built per app from `auth.fetch` and the gateway URL. The registry resolves a model by matching it against each candidate's discovered model ids, custom before hosted, where every candidate carries its own resolver thunk (a custom connection closes over `resolveConnection`; hosted closes over the injected transport), so resolution never branches on what a candidate is.
- **Non-static auth is also an injected transport, never a field.** Anything that cannot be serializable static data (the hosted session fetch; a future signing or token-refresh auth such as Bedrock SigV4 or Vertex OAuth) is composed by the caller into a `{ fetch, baseURL }` and injected. Its persistable secret, when it has one, lives in its own provider-shaped device-local store and a boot-time factory turns it into the injected transport; it does not enter `Connection`. The data layer stays two-tier (a serializable `Connection` can never hold a `fetch`); the resolved layer is the single unification point where heterogeneous sources become one `ResolvedConnection`.
- **A model id served by more than one connection resolves to a custom connection before hosted.** An explicit bring-your-own key wins the collision over the metered gateway, so a shared id is never silently billed to Epicenter credits. (Landed: `packages/app-shell/src/inference-picker/connections.svelte.ts`.)
- **The display label is derived from the base URL, not stored.** A connection matches a shipped preset by its full normalized base URL (so a self-hosted proxy that merely shares a host does not false-match, and Ollama's `:11434` stays distinct from LM Studio's `:1234`); otherwise the label is the URL host. Derived means it cannot drift when the user edits the URL, so the `preset` field is dropped.
- **Capability metadata is read-side decoration that never gates.** A discovered model is an id plus optional fields (`label`, `credits`, modality, tool support), sourced from the endpoint's own `/v1/models` extra fields or a later build-time-vendored catalog, kept in the registry's side cache, not on the connection. Hosted's product `label`/`credits` are these optional fields. A model with no metadata stays selectable.
- **The boundary is unchanged from [ADR-0059](0059-an-inference-connection-is-a-capability-orthogonal-device-endpoint.md).** The connection owns where and how to authenticate; it composes with, and does not absorb, the per-capability operation engine (chat, transcription, embeddings: path and body differ), the downloaded in-process engine ([ADR-0022](0022-rust-owns-the-models-folder-the-webview-owns-the-catalog.md)), and signing auth.

### Two widenings are named and deferred, both non-breaking to add

The flat pair is the floor, not a ceiling. The review found exactly two future needs and confirmed each is a non-breaking, additive change, so neither is built until it has a live producer or consumer:

1. **A static `headers?: Record<string,string>`** for the cases that are static data but not a lone Bearer: AI gateways that need a Bearer *and* a second header at once (Helicone's `Helicone-Auth`, Cloudflare's `cf-aig-authorization`, Portkey's `x-portkey-api-key`), Azure's `api-key` header on its `/openai/v1/` route, and OpenRouter attribution. **Trigger to add:** a real request to drive one of those through a bring-your-own connection. Adding it is the type, the resolver (set the Bearer first, then spread headers so they win on collision, matching the Vercel AI SDK merge order), the arktype schema, and an "Advanced" disclosure in the connect form, as one coherent slice.
2. **More than one injected transport** (a labeled list rather than a single `hosted`), when a second non-static-auth source (Vertex, Bedrock SigV4) is actually wired. **Trigger to add:** that second injected source.

Both seams are deliberately *not* pre-built: a public field or a pluralized seam with no producer is structure that implies a completeness the product does not have.

## Consequences

- **The leak guard survives verbatim (ADR-0053).** `resolveConnection` takes only connection data and never an Epicenter fetch, so a custom turn is structurally incapable of carrying the session bearer; the one origin check still lives on the credential in `fetchWithAuth`. The two-arg `resolveConnection(connection, hosted)` is deleted, which also removes a dead `hosted` argument that `discover` and the smoke scripts threaded for nothing.
- **The same connection drives chat, transcription, and embeddings with zero reshape.** This is already half-built: Whispering shares one OpenAI endpoint config across chat (`/v1/chat/completions`) and transcription (`/v1/audio/transcriptions`) today. A future transcription consumer reuses the registry; path and body are the operation engine's, not the connection's. The two bespoke transcription providers (Deepgram's `Authorization: Token` plus raw body, ElevenLabs' `xi-api-key` plus `model_id`) are correctly the ones that do not fit the OpenAI wire and keep their own config.
- **Client I/O consumes the resolved transport, not the connection.** `resolveConnection` is the one boundary that turns a `Connection` into a `ResolvedConnection`, and the caller crosses it: every operation that does I/O (the chat engine, `listModels`, and the shared `transcribe` client in `@epicenter/client`) takes the resolved `{ fetch, baseURL }`, so it never re-resolves and never branches on whether the transport came from a third-party key or the injected hosted session fetch. A function that accepted a raw `Connection` would re-introduce the hosted-vs-custom branch this ADR deleted.
- **Coverage today is honest and wide.** The floor covers every OpenAI-compatible chat provider and the OpenAI-compatibility endpoints of Anthropic, Gemini, and Bedrock-with-an-API-key (all Bearer), plus keyless local servers. The dual-credential gateways and Azure's `/openai/v1/` route wait on the `headers` widening; Vertex OAuth and Bedrock SigV4 wait on a second injected transport. Named out of scope so nobody crams them in: classic Azure (a mandatory `api-version` query and a deployment in the path) and the native Bedrock/Anthropic/Gemini wires (non-OpenAI bodies, or a model in the path).
- **Soft data migration.** The persisted schema drops `kind` and `preset`. arktype retains the now-undeclared keys without error, so existing device-local connections keep working untouched; the resolver and the derived label ignore the stale keys, and the next `add` rewrites them clean. No reset, no clean break.
- **The hosted gateway does not need to expose `GET /v1/models`.** Because hosted is the injected transport with a per-app injected catalog, there is no hosted-specific model knowledge left in shared client code to erase, so the gateway-discovery route [ADR-0059] left open is unnecessary, not merely deferred.

## Considered alternatives

- **Keep the `kind` discriminant ([ADR-0059](0059-an-inference-connection-is-a-capability-orthogonal-device-endpoint.md) as-is).** Fine for five Bearer-or-keyless presets, but it forces a branch the moment a non-Bearer case appears and keeps hosted's `label`/`credits` as a special case rather than decoration.
- **A closed `auth: epicenter-session | bearer | header | none` union.** Rejected: it is exclusive, so it cannot express the real bearer-and-a-header gateways, and it re-encodes the hosted/custom split as `epicenter-session` plus an `editable: false` flag, which is the discriminant the collapse claims to delete. No comparable tool that shipped this primitive kept an auth-type union.
- **Ship the `headers?` bag now, before a consumer.** Rejected: nothing in the connect form produces it, so it would be a public field reachable only by hand-editing storage, implying a completeness the product lacks. It is non-breaking to add with its producer, so it is deferred with a named trigger rather than carried empty.
- **Fold `apiKey` into `headers` (one field, `headers: { Authorization: 'Bearer ...' }`).** Rejected: it collapses past the natural joint. `apiKey` is the secret credential; headers are request decoration; every comparable keeps them separate. A `headers`-only shape forces the form to parse `Authorization` back into a key field on edit, re-introducing exactly the derive-from-a-string fragility this ADR removes from the label.
- **Store the `preset` as a hint for the label.** Rejected: the resolver never reads it, and a stored hint drifts when the user edits the URL. Deriving the label from the base URL cannot drift.
- **Resolve hosted before custom on a model-id collision.** Rejected: it silently meters a user who brought their own key. Bring-your-own must win.
