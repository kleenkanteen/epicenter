## Context

The relay floor (ADR-0073, Invariant 5) carries one Epicenter-routed transport for both convergent sync and cross-device tool calls. Grounding the recurring user moment ("driving, on my phone, reach a tool that lives only on my Mac Studio at home") against the code surfaced three facts: the floor is built but has zero end-to-end consumers (its only would-be consumer, opensidian auto-mounting a `books` route, is wired to a producer ADR-0072 deliberately deferred); the wedge (Whispering) uses none of it and is not even wired into sync; and the floor's only unique property over an off-the-shelf overlay (Tailscale) is zero-install-browser access to a NAT'd box.

Two physical facts bound the design. A NAT'd home box accepts no inbound connection, so reaching it always requires a publicly addressed rendezvous both parties dial out to; CGNAT makes "home box, browser-first, no third party" impossible (even iroh and WebRTC fall back to a relay on hard NAT). And the data itself divides: most of what a device wants is already present as a local-first replica, while the one thing that genuinely needs live reach is a tool call against immovable home data (Local Books, which is never replicated to the phone). The forcing question: should one Epicenter-routed transport carry both data sync and capability calls, or are these two planes with different owners and different paths?

## Decision

Cross-device is two planes.

1. **The sync plane (Epicenter, hosted, cheap).** Epicenter Cloud syncs the CRDT (Yjs) across a user's devices over a plain authenticated WebSocket to a per-user, hibernatable anchor, gated on Epicenter sign-in. This is light, it is the paid product (the Obsidian-Sync analog), and Epicenter sees this data (the hosted-sync deal, ADR-0004/0068). Multi-device offline-merge holds now; multi-user collaboration lands here later. Yjs is the source of truth ONLY for genuinely multi-writer data; single-authoritative-writer data (Local Books mirrors QuickBooks) stays the box's SQLite, never Yjs.

2. **The capability plane (the user's box, direct, Epicenter NOT in the path).** A daemon on the user's always-on box (the Mac Studio) holds the heavy, immovable, sensitive resources: SQLite, local models, Local Books, actions. The user's devices reach it DIRECTLY over an overlay the user provides (Tailscale or a tunnel today; an embedded iroh or a rented broker only if turnkey box-access is later earned). The actual MCP `tools/call` flows device to box; Epicenter never routes it. This generalizes ADR-0072 (Local Books over Tailscale) into the single capability story. The box exposes its capabilities as URL-addressed surfaces, an OpenAI-compatible `/v1` for local models and an MCP endpoint for tools, reached as the same `{baseUrl, token?}` connection ADR-0078 defines for inference. Reachability is a pluggable overlay transport orthogonal to that surface: a tailnet address by default, an opt-in public tunnel or Tailscale Funnel URL when a browser must reach the box, and an embedded iroh NodeId only behind the turnkey trigger. A new transport is an added address, never a new protocol.

3. **Discovery rides the sync plane; calls ride the capability plane.** The daemon publishes the list of endpoints it can be reached at (each a `{kind, transport, baseUrl}`: a tailnet MCP and `/v1` URL by default, plus a public tunnel or Funnel URL when the box opts into browser reach) together with its MCP tool manifest, into a small synced "capabilities directory" doc. The super-chat agent loop READS that directory over the sync plane (light, Epicenter-brokered) and DIALS whichever endpoint its own transport can reach DIRECTLY (off the Epicenter path): a native app on the tailnet uses the tailnet URL, a browser uses a public URL or finds none and degrades. Publishing the address is safe because reachability is gated by the overlay's authorization (a Tailscale ACL, a tunnel token), not by address secrecy. A later transport (iroh) is one more entry, not a protocol change. Manual endpoint entry is the v0 fallback and the escape hatch for a box not on the sync plane.

4. **Inference stays orthogonal** (ADR-0078): a `{baseUrl, token?}` service, the hosted gateway (revenue) or the box's local models, never on either plane's transport.

**Capability and inference are one connection primitive.** A box surface (its `/mcp` tools, its `/v1` models) and a hosted or remote inference endpoint are the same `{baseUrl, token?}` connection (ADR-0078): the agent resolves a connection and calls a URL. What varies is only which transport resolves the URL and whether Epicenter is in the path (never for a box, optionally for hosted inference). The relay floor existed to BE a transport; once the box is a URL and the user's overlay is the transport, the floor carries nothing, which is why its capability layer collapses to deletable.

This refines ADR-0073 Invariant 5 ("one transport carries sync and tool calls alike"): sync and capability are now two planes with different owners. ADR-0073's other invariants (the MCP data vocabulary, the host-enforced read/write tier, read-only to foreign hosts, the OpenAI-compatible model boundary) stand.

## Consequences

- **Cost scales with the cheap thing.** Epicenter runs only the hibernatable sync anchor, never an always-on compute box; the expensive always-on box is the user's. Tiering follows: free = local-only or bring-your-own-sync; paid = hosted CRDT sync; metered = inference. Epicenter bears cost only where there is revenue.
- **The privacy "middle ground" dissolves.** Epicenter is in the path for sync (acceptable: that is the hosted-sync deal) and never for your box (your sensitive compute stays off the Epicenter path). Two clean ends, not a middle.
- **The relay floor's capability layer becomes deletable.** The in-room dispatch subsystem and the daemon-as-mesh-peer (`run --peer`) are already gone (PR #2237). What remains, deletable but frozen behind the Trigger, is the floor's channel layer: the 4-frame channel protocol and `channel-bytes`, the `channel-router`, the relay acceptor (`relay-channel/acceptor`, `daemon/open-relay-acceptor`), presence-as-tool-directory, and cross-device MCP carried over the floor. The plain CRDT-sync WebSocket and the anchor survive untouched: the binary y-protocols sync path rides the socket independently of the additive text-frame channel layer (`open-collaboration` exposes it only through `textPort` and never parses a channel frame), so deleting the channel layer cannot disturb sync.
- **Web reach is an opt-in box capability, not a global mode.** A browser reaches a box only when the box advertises a public URL (a tunnel or Funnel), because the browser is not on the tailnet; bare Tailscale serves native apps only. The default is native-on-tailnet (zero provisioning, the ACL is the authz); exposing a public URL is a deliberate posture change (public ingress makes a box token load-bearing), so it stays opt-in per box.
- **One unique property is given up: zero-install-browser access to a box.** Reaching a box now needs an overlay client (Tailscale) or an embedded transport (iroh), not a bare browser. For a developer that is setup they already do; for a non-developer it is friction.
- **The wedge is unblocked independently.** Whispering's real cross-device need is the sync plane (currently unwired), not the capability plane. The highest-leverage next build is wiring Whispering into sync, which this decision frees from any dependence on the floor.

## Trigger to revisit (the open product question that gates the deletion)

Whether the relay floor's capability layer is deleted or kept and productized turns on one product call, not an engineering one: **is "reach my box" a power-user feature (bring-your-own-Tailscale, Epicenter owns no capability transport, the floor dies) or a mass-market turnkey product (Epicenter provides reachability via an embedded iroh or a rented broker, the floor or its successor lives)?** The current lean is the power-user answer: Epicenter owns zero box transport and the user brings the overlay. So the freeze is bound, not open-ended: the floor's channel layer is deleted once Whispering is wired into the sync plane (the next build), UNLESS turnkey, no-setup box-access for non-developers has by then become a committed product goal. Until that milestone no further spend goes to the floor-extension or broker tracks, and embedding iroh or building a reachability broker returns only behind that same turnkey commitment.

## Considered alternatives

- **Keep one Epicenter-routed transport for sync and tools (the relay floor, ADR-0073 Invariant 5).** Rejected as the default: it has zero consumers, conflates two planes with different requirements, and keeps Epicenter in the path of your own box for no privacy or cost benefit.
- **Embed iroh now so box-access is turnkey without Tailscale.** Deferred: it re-introduces exactly the dependency ADR-0073 deleted, still needs a discovery service plus a DERP-relay fallback (the rendezvous is un-removable, only relocatable), and for the founder's own developer use Tailscale already suffices. It returns only behind the turnkey trigger above.
- **Drop Yjs for a simpler last-write-wins sync.** Rejected: even a single user is concurrent across their own offline devices (last-write-wins silently loses data on an offline merge), and collaboration-later is a stated goal. Yjs earns its keep on the sync plane, and its cost scales with collaborative data, not with the box.
- **Collapse the sync relay onto the Mac Studio (one box for sync and capability).** Rejected as the default: the relay must be publicly addressable and the box is NAT'd, so they pull onto different machines. They collapse only on a publicly addressed box (a VPS, or a tunneled box that accepts the exposure), which a self-hoster may choose but is not the default.
