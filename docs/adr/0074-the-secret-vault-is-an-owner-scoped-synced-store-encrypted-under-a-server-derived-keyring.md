# 0074. The secret vault is an owner-scoped synced store encrypted under a server-derived keyring, not a passphrase vault

- **Status:** Accepted
- **Date:** 2026-06-27
- **Amended by:** [ADR-0092](0092-identity-is-the-partition.md) for identity vocabulary and durable namespace bytes.
- **Amends:** [ADR-0054](0054-an-inference-backend-is-the-metered-gateway-or-a-custom-server.md)'s "a provider key is device-local, never synced." The rule was right against the evidence ADR-0054 had: a synced key on the plaintext relay is a secret in the clear, and a synced `localhost` URL is meaningless elsewhere. The evidence it never weighed is *cross-app*: two first-party apps (Whispering and Vocab today) each need the same provider key, and device-local storage cannot cross an app or origin boundary. Re-entering your Groq key in every app is the exact friction the "own your data, apps are lenses" mission exists to kill. So a `bring`-class key syncs through the vault; a device-scoped endpoint (`localhost`, LAN) still stays device-local, which keeps ADR-0054's real concern intact.
- **Supersedes:** PR #2056's three vault ADRs as they exist on `feat/secret-vault-encrypted-kv` (numbered 0053 / 0054 / 0055 on that branch: one-home-at-a-time, owner-scoped persistence, and passphrase-as-the-only-key-source). Those numbers are already taken on `main` by unrelated decisions; the vault ADRs were renumbered three times (0004/0005, then 0041/0042/0043, then 0053/0054/0055) chasing collisions, and that thrash ends here. The one-home and owner-scoping decisions survive as invariants 4 and 2 below; the passphrase-key-source decision is the one this reverses.

## Context

Epicenter data splits two ways, and the split decides what the vault holds. Things you **make** inside an app (recordings, transcription runs, saved tabs, notes) are byproducts of one app doing its job; they stay private to that app. Things you **bring** (your accounts, your API keys, your credentials) are yours, exist before any app, and every app should read them. The vault is the home for the second category. It is a user-level asset that apps are lenses over, not an app feature.

PR #2056 built the foundation but bet the key source on a user passphrase: Argon2id over a typed passphrase derives a key-encryption key that unwraps a random master key, so the relay stores ciphertext the operator cannot read even on hosted. That is real zero-knowledge, and it carries the real cost zero-knowledge always carries: an unlock ceremony, a `locked` state every consumer must branch on, and a lost-passphrase cliff with no recovery. The forcing question was whether server-blindness against the operator is the vault's moat or a nice-to-have. It is a nice-to-have, and ADR-0068 already answered how a user who wants it gets it: self-host, become the server, never an in-app passphrase. A passphrase vault would tax every consumer and every new-device flow to buy a property the deployment choice already provides.

## Decision

The vault is one owner-scoped, user-global synced document holding only what a user brings, encrypted at rest under a server-derived per-owner keyring delivered on the authenticated session. No passphrase, no unlock, no `locked` state. Five invariants:

1. **Encryption keys to the owner, not a passphrase.** The synced vault's values are encrypted at rest under a server-derived per-owner keyring (`HKDF(rootSecret, ownerId)`, or a random per-owner key wrapped under the deployment root) delivered over the authenticated session: the same posture every other confidential synced value already uses (the body-docs server-derived keyring, not zero-knowledge). The relay stores ciphertext and never holds the keyring (it lives in the auth service, not the relay DO), so a relay or backup leak alone does not expose keys. That is defense in depth, not a privacy claim. On hosted the operator can read a stored key, and that is the accepted point: privacy from the operator is achieved by becoming the server, where self-host derives the per-owner key with *your* root and Epicenter is never in the loop, per ADR-0068/0069/0070, never by client-side crypto (which ADR-0004 rejected). This is why there is no passphrase, no unlock, and no lost-passphrase cliff.

2. **Owner-scoped, not guid-scoped.** The vault keys to its owner in the persistence layer; the guid is a stable constant. Per-user partitioning comes from the owner, which is exactly what makes one shared doc safe across apps. (This is PR #2056's owner-scoping decision, kept.)

3. **One shared doc, user-global.** One vault per user, addressed by a stable user-global guid, attached by every app. Not one vault per app. Promoting PR #2056's per-app guid to a user-global constant is a constant change, not a redesign, because invariant 2 already makes it safe.

4. **One home per secret, no two-place resolver.** A secret is *either* synced *or* device-local; the facade never reads from two places and merges. Stated per-secret, not whole-vault: most keys sync, but a single key may be pinned device-local (the work-laptop case) by choosing its home, not by layering a local override on a synced default. "Synced default plus local override" is the resolver this forbids; "this key lives on this device" is the allowed home choice. (This is PR #2056's one-home decision, kept.)

5. **One read contract for every consumer: `available | missing`.** `missing` means no key, remedy is add it; `available` means decrypted and usable. With a server-derived keyring there is no `locked` state: an authenticated session has the keyring, so a present key is always decryptable. Consumers never call a provider SDK with a blank key; they branch on this contract.

The passphrase-derived keyring (true zero-knowledge while hosted) is the deferred premium seam, the exact one ADR-0068's considered-alternatives names: added only if a "trust you with my data but not my API key, and will not self-host" segment becomes real, as an opt-in upgrade on top, never the default. PR #2056's encrypted-KV primitive (`createEncryptedLwwKeyValue`, `@epicenter/encryption`) is key-source agnostic and survives unchanged; only its passphrase keyring (Argon2id, wrapped master key, passphrase strength, `generatePassphrase`) is shelved until that trigger fires.

ADR-0092 later renames the live partition vocabulary from owner to principal and changes the HKDF info label from `owner:${label}` to `principal:${label}` under the zero-durable-data clean break. This ADR's product decision still stands: synced secrets key to the authenticated partition and use a server-derived keyring, not a passphrase.

## Consequences

- **The vault lifecycle collapses to a per-secret home toggle.** With no passphrase there is no provisioning ceremony: enabling sync on a secret is instant because the authenticated session already holds the keyring, `disableSync` migrates a synced value back to device-local before detaching, and `forget()` simplifies to "remove from this device," a local replica wipe that never propagates deletes. There is no lost-passphrase recovery path to model, because there is no passphrase to lose; re-signing in re-derives the keyring and the synced secrets return.

- **New-device and cross-app reads are immediate.** Sign in, the owner KV syncs, the keyring arrives with the session, and keys read `available` with no unlock. Cross-app reuse needs nothing shared between apps: each app derives the same per-owner keyring from its own authenticated session, so Whispering and Vocab decrypt the same vault even on different origins. Same-origin and path-routing stay a nice-to-have for auth simplicity, no longer a hard requirement for cross-app secrets.

- **The honest cost is the trust line, stated where keys are entered.** On hosted, a synced provider key is operator-readable. This is ADR-0068 applied to secrets, and the gateway model (ADR-0054) already means most hosted users store no bring-your-own key at all. The remedy for a user who refuses that is self-host, not a passphrase. UI copy says "Synced to your devices" versus "Saved only on this device," never the word "plaintext," and there is no recovery-code ceremony to ship.

- **First-party apps share all vault credentials by design.** It is the same user's key regardless of which app reads it, so there is no leak, only the intended convenience. Because the key now appears in a second app automatically, the save-time announcement ("your other apps can use it too") matters more, not less. If third-party apps ever exist, a per-app capability grant seam is added then; it is not built now, because that consumer does not exist.

- **PR #2056 reshapes rather than reverts.** Wave 1 (the encrypted-KV primitive and `@epicenter/encryption`) survives whole; wave 2 (the passphrase keyring) and the vault lifecycle (unlock/lock/forget-as-recovery) are dropped; the facade's read union collapses to `available | missing` at its four call sites and in `ProviderConfigFields`; the guid is promoted to user-global; and the key source is wired to the server-derived keyring behind the auth seam. Whispering has no auth yet, so the no-account path degrades to device-local plaintext (the safe degenerate), and the server-derived keyring activates when auth lands.

- **What this forecloses:** a passphrase as the default key source, and a `locked` read state, permanently. Re-introducing either is re-litigating this ADR. The only key-source ladder is "server-derived by default, passphrase as a deferred opt-in premium seam," and the only privacy move is relocating the anchor (self-host), not flipping a setting.

## Considered alternatives

- **Passphrase-derived keyring as the default (PR #2056's bet).** Rejected as the default, kept as a deferred seam. It buys zero-knowledge against the hosted operator at the cost of an unlock ceremony, a `locked` state on every consumer, and a lost-passphrase cliff, to serve a "trust you with my data but not my key, and will not self-host" segment that is not yet real. ADR-0068 already routes the privacy-from-operator user to self-host, so the passphrase earns its complexity only when that narrow segment asks for it.

- **Keep provider keys device-local per ADR-0054.** Rejected on the cross-app evidence ADR-0054 never weighed: device-local storage cannot cross an app or origin boundary, so every app re-collects the same key. The device-local rule survives for what it was actually protecting, the device-scoped endpoint (`localhost`, LAN), which stays device-local while its key syncs.

- **One vault per app.** Rejected: it re-collects the same brought credential per app and defeats the lens model. Invariant 2's owner-scoping already makes a single user-global doc safe, so the per-app guid was incidental, not load-bearing.

- **A synced-default-plus-local-override resolver for the work-laptop case.** Rejected: it is the two-place read invariant 4 forbids. The same need is met by choosing a secret's single home, not by layering a local read over a synced one.
