# 0105. Vocab is a multilingual tutor and readings are a client-side derived view

- **Status:** Accepted
- **Date:** 2026-07-03
- **Relates:** [ADR-0102](0102-vocab-stores-verbatim-terms-under-a-human-owned-note-and-refuses-glosses-srs-and-provenance.md) (verbatim, language-blind terms: this reinforces the no-language-column refusal)

## Context

Vocab shipped Chinese-only: a bilingual Chinese-English system prompt (`VOCAB_SYSTEM_PROMPT`) and a single pinyin romanizer keyed to the Han Unicode range. Repositioning Vocab as a general-purpose multilingual tutor forced a decision about how pronunciation readings (pinyin, romaji, and so on) should work across scripts, under a hard local-first requirement: the app can be opened or compiled to run against any OpenAI-compatible endpoint, including a local one, and reviewing your own saved terms and past lessons must work offline.

The design space, explored over several passes: emit reading markup inline in the tutor's answer; annotate a clean answer after the fact with a lazy AI pass; build a per-script deterministic romanizer library behind a script-detection router; or ship a user-installable romanizer plugin catalog like the runtime model download in Whispering and Handy. Two facts settled it. First, most studied languages are Latin-script and need no readings at all; the burden is a small, known, slow-changing set of non-Latin scripts. Second, the deterministic romanizers for those scripts are pure-JS npm libraries of KB to low-MB size (pinyin-pro sub-MB; the outlier, the Japanese kuromoji dictionary, ~12MB), not the 40MB-3GB native ML weights that justify Whispering's runtime-download catalog.

## Decision

- **The tutor is language-blind.** One general multilingual system prompt: it infers the studied language from the conversation and may intermix languages. There is no target-language setting and no per-term language column. ADR-0102's no-language-column refusal holds and is reinforced: the studied language is already in the text for the only mechanical consumer (readings, which read the script), and grouping the pool by language is the decks/tags machinery ADR-0102 refused.
- **Readings are a client-side derived view over clean text, never content and never conversation memory.** The tutor writes plain prose; the assistant message is stored verbatim; readings are rendered on top through the existing `Romanizer = (text) => Segment[]` and `<ruby>` seam in `@epicenter/ui/markdown`, and the toggle hides them for free. Emitting reading markup inline in the answer is rejected: the message text is fed back to the model as memory on later turns, so display markup would poison future context and force stripping at the terms, harvest, and prompt boundaries.
- **Readings are produced by a small ordered registry of deterministic, per-script romanizers, lazily loaded on first use.** `pinyin-pro` (Chinese) is the first and flagship provider. A provider owns one script, is pure and synchronous, decides cheaply whether a passage contains its script, and dynamic-imports its library only when that script is first read (a code-split chunk the browser caches). Providers are disjoint by script and composed, so a mixed sentence is covered run by run. This is fully offline, deterministic, and free of any model call, network round-trip, non-determinism, or validation. The AI's role is to teach (generate the answer); it never annotates readings.
- **Scripts fall into three reading categories, and each has exactly one sanctioned path.** The dividing line is not "phonemic vs not" (Hangul is phonemic yet lands in category 2); it is "does a cheap offline package hit a self-consistent, correct target".
  1. **Cheap and correct: deterministic, shipped.** The script carries the reading and a pure-JS library delivers it offline. Lexically complete Han (`pinyin-pro` ships the per-character dictionary), the kana syllabary (`wanakana`), and near-1:1-phonemic Cyrillic (`transliteration`, a standard orthographic transliteration). Deterministic wins outright; an AI pass here would be strictly worse, trading a network dependency and per-message cost for no quality gain.
  2. **Hard but specifiable: deferred; the earned-later path is a composing online AI annotation, not a hand-built romanizer.** The reading is recoverable in principle, but only with lexical or morphological knowledge that no lightweight offline package delivers reliably: Korean Revised Romanization (cross-syllable assimilation plus morpheme boundaries), Japanese kanji (`kuroshiro` + `kuromoji`, ~12MB dict), and Modern Greek γ (no consistent off-the-shelf phonetic engine). These get no automatic reading today. When a real audience appears, the sanctioned move is not to hand-build a deterministic engine but to add the one AI-annotation provider described below, which claims exactly these scripts when the app is online.
  3. **Under-specifying: no deterministic reading is possible; prose, and optionally AI annotation.** The writing system omits the information, so no rule can recover it: abjads (Arabic, Hebrew) drop short vowels, Devanagari deletes the inherent schwa by context, Thai writes no word boundaries and drops tone. The tutor's prose explains the sound; the AI-annotation provider is the only possible reading source.
- **No AI reading path, no user-facing provider control, no plugin catalog, in the shipped design.** Providers are disjoint by script, so dispatch is automatic and multiple providers are always active together; there is nothing for a user to toggle or reorder, and a provider-priority UI would leak implementation. The only control is Show/Hide readings. The one genuine overlap, Han shared by Chinese and Japanese, is handled by the pinyin provider abstaining when kana or hangul is present; if it ever needs resolving it is a "Read Han as: Chinese / Japanese" language preference, never a provider list. An AI-annotation provider is a deliberately refused, earned-later single registry entry: the sanctioned path for both the hard-but-specifiable (category 2) and under-specifying (category 3) scripts, added only when such an audience becomes real. It always runs post-hoc on the already-clean message, so it never poisons memory the way inline markup would; it is gated on connectivity; and it composes with the deterministic registry rather than replacing it, claiming only the scripts the deterministic providers abstain on while they stay the offline floor for Han, kana, and Cyrillic. It never becomes the offline floor itself, because that would forfeit the local-first payoff.

## Consequences

- The reading overlay never depends on an inference endpoint: it works fully offline, for saved terms and past lessons, against any or no backend. This is the local-first payoff, and it is why `pinyin-pro` is kept and load-bearing rather than deleted.
- Readings are deterministic, so the failure mode is a missing reading, never a wrong one. There is no round-trip validation, no per-message model call, and no reading cache to manage; the pinyin provider's kana/hangul guard is the only safety rule.
- `@epicenter/ui/markdown` is unchanged except one additive export (`identityRomanizer`, the passthrough, so a consumer can inject a romanizer without a branch). The `Romanizer`/`Segment`/`<ruby>` seam serves an ordered set of deterministic providers exactly as it served the single pinyin romanizer; the registry and providers live in the app, where language knowledge belongs, not in the shared UI package.
- The committed short list, all deterministic, pure-JS, and free of any runtime dictionary download: Chinese (`pinyin-pro`), Japanese kana (`wanakana`), and Cyrillic (`transliteration`) are shipped. Korean and Greek were candidate "cheap adds" and were evaluated against the invariant that a wrong reading is worse than a missing one; both were deferred (see Considered alternatives). Japanese kana reads は as `wa` but leaves kanji unread: furigana over kanji needs a morphological analyzer (`kuroshiro` + `kuromoji`, ~12MB dict) and is deferred; it is the one heavy case and would load its dictionary on demand.

## How to add a language (next steps)

1. Write a `ReadingProvider` in `apps/vocab/src/lib/readings/` (see `pinyin.ts`): a `matches(text)` script test and a `load()` that dynamic-imports the library and returns a `Romanizer`.
2. Add it to the ordered `readingProviders` array in `registry.ts`. That is the whole change; the resolver, the composition, and the `<Markdown>` seam are untouched.
3. Keep the disjoint-by-script discipline. The only ordering that matters is the Chinese/Japanese Han overlap, which the pinyin provider already sidesteps by abstaining on kana/hangul.
4. Ship the standard romanization system per script by default (pinyin with tones, Hepburn, Revised Romanization); add a per-script system picker only if a user asks. Only add a deterministic provider for a category-1 script (a cheap offline package hits a correct target). For a category-2 (hard-but-specifiable: Korean, kanji, Greek) or category-3 (under-specifying) script, do not hand-build a romanizer or ship a plugin catalog; the earned-later path is the single AI-annotation provider, post-hoc and online-gated, composing with the deterministic floor, added only when that audience is real.

## Considered alternatives

- **Inline reading markup in the generated answer.** Rejected: assistant messages are conversation memory fed back to the model, so display markup contaminates future context and forces stripping at the terms, harvest, and prompt boundaries.
- **A lazy, round-trip-validated AI annotation as the shipped reading producer.** Built and then removed. It reaches any script with no library, but it makes every reading depend on an inference endpoint (anti-local-first: no readings offline, and none from a weak local model, which fails validation and renders clean), and it is non-deterministic, needing a per-message call, a cache, and "omit when unsure" prompt discipline. What was removed is AI as the *shipped, sole, always-on* reading producer. What is reserved is AI as an *optional, online-gated, post-hoc enhancement layer that composes over* the deterministic floor: for under-specifying scripts (category 3), where a deterministic reading is impossible, and for hard-but-specifiable scripts (category 2: Korean, kanji, Greek), where a correct reading needs lexical or morphological knowledge no lightweight offline package has. The composing form keeps the local-first payoff intact (Han, kana, and Cyrillic still read offline) while letting the hard tail read when online.
- **A user-installable romanizer plugin catalog (Whispering/Handy style).** Rejected: that machinery (a native models folder, staged download with an integrity floor, an OPFS/IndexedDB blob store, install/uninstall state and UI, versioning, plugin security) exists for 40MB-3GB native ML weights. Deterministic romanizers are KB to low-MB pure-JS libraries, so lazy dynamic import plus browser caching is the right delivery; a catalog would be a second package manager inside the app for a set of about a dozen known scripts.
- **Per-language local libraries behind a script-detection router, or a stored per-term language column.** Rejected: the router grows a permanent library-per-script family plus a kana/precedence detector, and the column is the decks/tags grouping ADR-0102 refused, whose only consumer (readings) already reads the script from the text.
- **A Korean Hangul provider (Revised Romanization).** Deferred. Correct RR is defined by cross-syllable assimilation (신라 -> `silla`, 좋다 -> `jota`, 종로 -> `jongno`), and the lightweight deterministic packages get it wrong on common words: `hangul-romanization` does naive per-syllable mapping (신라 -> `sinra`); the better `@romanize/korean` and `aromanize` each still miss high-frequency cases and disagree with each other (종로 -> `jongro` in both, `별내` split between them). Faithful RR needs morpheme-boundary knowledge, i.e. an analyzer with a dictionary, which puts it in the same class as Japanese kanji, not the "cheap add" class. This is a category-2 script (hard but specifiable): its earned-later path is the composing online AI annotation, not a hand-built deterministic engine. A wrong reading violates the core invariant, so no Korean reading beats a wrong one.
- **A Greek provider.** Deferred. There is no clean deterministic win. Orthographic transliteration (the generic `transliteration`, or `greek-utils.toTransliteratedLatin`) misleads on the stop-cluster digraphs that define Modern Greek pronunciation (μπύρα -> `mpyra`, not `bira`; ντομάτα -> `ntomata`, not `domata`). The phonetic romanizer `greek-utils.toPhoneticLatin` gets those digraphs and every vowel digraph right (αι->e, ει->i, αυ->af, ευ->ef) but is unreliable on γ, a high-frequency consonant: it is inconsistent before front vowels (γεια -> `gia` but γελάω -> `yeláo`) and systematically wrong on the γγ/γκ nasal clusters (άγγελος -> `agyelos`, not `angelos`; Αγγλία -> `Agglia`, not `Anglia`). Because a provider reads every run of its script and cannot omit per word, that ships unpredictable wrong readings on common words. This is a category-2 script (hard but specifiable): revisit either if a purpose-built, consistent Modern-Greek phonetic library appears (a category-1 promotion), or via the composing online AI-annotation provider when a Greek audience is real.
